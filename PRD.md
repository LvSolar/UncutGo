# FindMyMovie PRD（初稿）

## 1. 项目一句话

用户输入一部电影，系统自动聚合豆瓣与国内播放源信息，判断“去哪里更可能看到无删减版”。

## 2. 背景与问题

当前手动流程是：

1. 去豆瓣查看评分、片长和基础信息。
2. 查看该电影在哪些国内平台可播放。
3. 逐个平台核对片长是否与豆瓣一致。
4. 根据片长差异，猜测哪个版本更可能是无删减版。

这个流程耗时、重复，而且容易误判。

## 3. 核心目标

### 用户目标

- 输入电影名，快速得到“去哪看更可能是无删减版”。
- 同时看到导演、豆瓣评分、版本片长、候选播放源、各播放源片长、判定理由。

### 产品目标

- 把“搜索电影 + 查播放源 + 对比片长 + 给出结论”自动化。
- 降低因为分钟取整、版本差异、片头片尾误差造成的误判。

## 4. 不做什么

- 不判断画质、字幕质量、码率。
- 不提供盗版资源聚合。
- 不承诺 100% 法律或发行层面的“官方无删减版”结论，只输出“高概率判断 + 依据”。
- 不在第一版支持电视剧、综艺、剧场版合集。

## 5. 用户故事

### 核心用户故事

作为一个在意电影完整性的用户，
我想输入电影名，
这样我就能马上知道国内哪些平台更可能提供未删减版本，
并看到这个判断是基于什么得出的。

### 典型输出

- 豆瓣：`In Bruges (2008)`，片长 `107 分钟`
- 导演：`马丁·麦克唐纳`
- 可播放平台：腾讯视频、咪咕视频、优酷、爱奇艺、哔哩哔哩
- 平台片长：
  - 腾讯视频：`1:47:12`
  - 优酷：`1:45:51`
  - 爱奇艺：无法获取
- 判定：
  - 腾讯视频：`大概率无删减`
  - 优酷：`疑似删减`
  - 爱奇艺：`信息不足`

## 6. 产品形态建议

## 6.1 推荐顺序

第一阶段推荐做：**网页应用**

原因：

- 输入电影名直接出结果，最符合你的目标。
- 结果页可以很自然地展示“豆瓣信息 + 平台对比 + 判定理由”。
- 后续既能做账号系统，也能做缓存、历史记录、人工纠错。

第二阶段可扩展：**浏览器插件 / 用户脚本**

原因：

- 你本来就经常在豆瓣看片页。
- 插件可以在豆瓣页面直接插入“去哪里看无删减版”的结果卡片。
- 这会非常顺手，但更适合作为网页产品成熟后的入口增强。

不建议作为最终主形态：**Codex skill**

原因：

- skill 更像开发辅助能力，不适合作为普通用户产品。
- 它不适合承载公开可用的搜索、展示、缓存、账号、分享链接等能力。

### 结论

建议路线：**网页应用优先，后续补一个豆瓣页插件/脚本作为快捷入口。**

## 7. 第一版范围（MVP）

### 输入

- 电影名
- 可选：年份
- 可选：豆瓣电影链接

### 输出

- 豆瓣电影基本信息
- 导演
- 豆瓣评分
- 豆瓣片长与版本信息
- 国内可播放平台列表
- 每个平台可获取到的片长
- 最终判定标签
- 判定理由

### 判定标签

- `大概率无删减`
- `可能无删减`
- `版本不一致`
- `疑似删减`
- `信息不足`

### 判定标签定义建议

- `大概率无删减`：平台时长与当前参考版本高度接近，差异处于正常误差范围内
- `可能无删减`：平台时长略短，但仍可能由分钟取整、片头片尾差异或展示误差导致
- `版本不一致`：平台时长与当前主参考版本不一致，但更像是另一个合法发行版本，而不是单纯删减
- `疑似删减`：平台时长明显短于参考版本，且没有证据表明它只是另一个正常版本
- `信息不足`：无法稳定拿到真实片长，或拿到的数据无法确认是否为正片

## 8. 核心流程

1. 用户输入电影名，或直接粘贴豆瓣电影链接。
2. 系统定位目标电影并确认唯一身份。
3. 系统抓取豆瓣信息。
4. 系统提取该电影可见的多个版本片长信息。
5. 系统选择当前主要参考版本。
6. 系统抓取豆瓣“在哪儿看这部电影”中的平台列表。
7. 系统逐个平台尝试获取片长。
8. 系统把参考版本片长与平台片长统一成秒。
9. 系统基于规则打标签并输出理由。

## 8.1 输入策略建议

第一版建议同时支持两种输入：

- 主输入：电影名
- 备用输入：豆瓣电影链接

### 原因

- 电影名最符合直觉，用户体验最好
- 豆瓣链接几乎零学习成本，也能作为搜片失败时的稳定兜底
- 两种输入并存，可以兼顾体验与准确率

### 默认处理逻辑

- 如果用户输入的是豆瓣链接，直接解析目标电影
- 如果用户输入的是电影名，先搜索候选电影，再做消歧
- 如果电影名匹配结果不够确定，提示用户补充年份，或直接粘贴豆瓣链接

## 9. 电影身份确认规则

为了避免搜错电影，系统需要尽量拿到“稳定标识”。

### 首选字段

- 中文名
- 原名
- 年份
- 导演
- IMDb 编号（如果豆瓣页可得）

### 原则

- 如果同名电影存在多个年份，优先让用户二次确认。
- 后续抓平台信息时，尽量使用更稳定的外部标识，而不是只靠中文名模糊搜索。

### 第一版匹配策略建议

- 电影名搜索结果只有一个高置信候选时，直接进入结果页
- 当存在多个同名或近似结果时，展示一个简短候选列表让用户点选
- 如果用户已经提供豆瓣链接，则跳过搜索与消歧

## 9.1 版本信息提取要求

系统应尽量识别并展示电影的多个合法版本，而不是只保留一个片长。

优先关注：

- 导演剪辑版
- 电影节版本
- 国际版 / 海外发行版
- 中国大陆版本
- 其他明确标注的修复版、加长版、重映版

## 10. 片长判定规则（非常关键）

### 10.0 参考版本选择原则

电影可能存在多个合法版本，例如中国大陆版、导演剪辑版、电影节版、国际版、加长版。

系统需要把这些版本尽量并列展示，并明确“当前用哪个版本作为主要参考”。

#### 对比优先级

判断平台是否疑似删减时，优先使用以下版本作为参考：

1. 导演剪辑版
2. 电影节版本
3. 非中国大陆版本
4. 其他明确标注的国际发行版本
5. 中国大陆版本

#### 同优先级冲突处理

如果同时存在多个高优先级版本，例如两个电影节版，或导演剪辑版与电影节版都被认为是高价值参考，则默认选择**时长更长的版本**作为主要参考。

这样更符合产品目标，因为本项目的核心诉求是优先帮助用户找到更完整的版本，而不是优先贴近中国大陆公映口径。

同时系统仍应展示：

- 被选中的主要参考版本
- 其他候选高优先级版本
- 它们之间的时长差异
- “当前按更长版本优先”的说明

#### 设计原因

- 中国大陆版本更可能因为审核或发行差异而较短
- 如果把中国大陆版本作为首要基准，容易把删减版误判成正常版本
- 优先参考导演剪辑版、电影节版、非大陆版本，更符合“尽可能找完整版本”的目标

#### 输出要求

- 结果页展示当前采用的参考版本
- 同时展示其他已识别版本及对应时长
- 如果多个高优先级版本之间差异明显，系统提示“版本较多，结论需谨慎”

### 10.1 统一单位

所有片长先转成秒再比较。

如果只拿到了豆瓣这种“按分钟展示”的片长，则应把它视为模糊值，而不是绝对精确值。

### 10.2 误差容忍

分钟展示会有取整问题，所以不能只按“分钟数字不同”就判定删减。

例如：

- 豆瓣 `128 分钟`
- 实际片源 `127 分 20 秒`

这不应判定为删减，因为差异只有 `40 秒`，很可能只是豆瓣按分钟展示。

#### 分钟展示值的理解方式

例如：

- 实际时长 `1:23:20`
- 豆瓣展示 `84 分钟`

这说明豆瓣分钟值更像展示口径，而不是精确到秒的元数据。

因此第一版建议：

- 如果有精确时长，优先按精确秒数比较
- 如果只有豆瓣分钟值，则允许更宽松的误差解释

### 10.3 初版规则建议

- 若平台片长比豆瓣片长短 `<= 90 秒`：先视为**正常误差范围**
- 若平台片长比豆瓣片长短 `> 90 秒 且 <= 5 分钟`：标记为**可能删减**
- 若平台片长比豆瓣片长短 `> 5 分钟`：标记为**疑似删减**
- 若平台片长比豆瓣片长长很多：优先怀疑是**版本不同**，例如导演剪辑版、加长版、片尾花絮版、重映版

这个阈值只是初版假设，后续可以按你的经验调整。

这里的“豆瓣片长”在实现时应理解为“当前选定的参考版本片长”，不一定是页面上最显眼的单一值。

### 10.3.1 第一版标签阈值建议

- `大概率无删减`：平台时长比参考版本短 `<= 90 秒`
- `可能无删减`：平台时长比参考版本短 `91 到 180 秒`
- `版本不一致`：平台时长比主参考版本长 `> 90 秒`，或明显更接近另一个已知合法版本
- `可能删减`：平台时长比参考版本短 `181 到 600 秒`
- `疑似删减`：平台时长比参考版本短 `> 600 秒`
- `信息不足`：无法获取可靠时长，或拿到的内容无法确认是不是正片

### 10.3.2 判定决策表

1. 如果拿不到平台真实片长，输出 `信息不足`
2. 如果拿到的平台内容疑似预告、花絮、合集片段，也输出 `信息不足`
3. 如果平台时长与主参考版本差异 `<= 90 秒`，输出 `大概率无删减`
4. 如果平台时长比主参考版本短 `91 到 180 秒`，输出 `可能无删减`
5. 如果平台时长虽然和主参考版本差异较大，但更接近另一个已知合法版本，输出 `版本不一致`
6. 如果平台时长比主参考版本短 `181 到 600 秒`，输出 `可能删减`
7. 如果平台时长比主参考版本短 `> 600 秒`，输出 `疑似删减`

### 10.3.3 为什么第一版采用 90 秒安全线

- 可以覆盖豆瓣分钟展示的取整误差
- 可以覆盖片头厂标、平台片尾处理等常见差异
- 不至于把真实几分钟级别的删减误放过去

### 10.4 不应直接视为删减的情况

- 秒级差异
- 片头厂标差异
- 片尾字幕裁切
- 平台展示值为向上/向下取整分钟
- PAL/NTSC 或不同发行介质引起的轻微时长差

## 11. 容易误判的边界情况

### 多版本问题

同一电影可能存在：

- 院线版
- 导演剪辑版
- 加长版
- 重映修复版
- 节展版
- 中国大陆版
- 国际版

如果豆瓣展示的是一个版本，而平台上架的是另一个合法发行版本，不能简单地说平台“删减”。

因此系统需要尽量输出：

- 已识别的版本名称
- 每个版本对应时长
- 当前使用哪个版本作为对比基准
- 为什么选择这个基准

### 平台信息问题

- 有的平台展示的是卡片页时长，不一定等于真实播放时长。
- 有的平台电影详情页写的是分钟，播放器里才有精确秒数。
- 有的平台需要登录或 VIP 后才能进入播放器并读取真实时长。
- 有的平台会把预告、正片、花絮混在一起，可能误抓错内容。

### 豆瓣信息问题

- 豆瓣片长未必总是最精确来源。
- 豆瓣只显示一个分钟值时，可能丢失秒级信息。
- 豆瓣有时会列多个片长版本，后续应优先保留全部版本信息。

## 12. 数据来源建议

### 主参考源

- 豆瓣电影详情页

用途：

- 电影身份确认
- 评分
- 片长
- 播放源入口

### 交叉参考源

- IMDb
- TMDb
- 其他公开电影数据库

用途：

- 交叉验证年份、原名、版本信息、片长
- 辅助判断“平台短了”还是“版本不同”

## 13. 技术路线建议

## 13.1 MVP 架构

- 前端：网页搜索页 + 结果页
- 后端：电影查询、平台信息聚合、判定逻辑
- 采集层：网页抓取 / 页面解析 / 必要时浏览器自动化
- 缓存层：缓存电影信息和平台片长，减少重复请求

## 13.2 为什么需要浏览器自动化

很多播放平台是前端渲染或播放器内才暴露真实时长，纯静态抓取不一定够。

所以更现实的方案是：

- 先尝试抓详情页公开数据
- 抓不到时，再退到浏览器自动化

## 13.3 第一版建议不要做的事

- 不要一开始就支持所有平台
- 不要一开始就做登录态托管
- 不要一开始就上复杂推荐算法

先把 2 到 3 个平台打通更重要。

## 13.4 技术栈定案（当前建议）

基于“主要给自己和朋友使用、不追求一开始就面向大众”的目标，第一版建议采用：

- 前端：`Next.js`
- 语言：`TypeScript`
- UI：`Tailwind CSS`
- 后端：`Next.js Route Handlers`
- 页面抓取：`Playwright`
- 页面解析：`Cheerio`
- 数据存储：第一版先不上数据库，优先使用本地文件缓存
- 运行环境：先本地运行，后续如有需要再部署到云端

### 为什么这样选

#### Next.js

- 一套工程里同时做页面和后端接口，适合个人项目快速推进
- 后续如果要部署给朋友用，迁移成本也低
- 做搜索页、结果页、接口层都比较顺手

#### TypeScript

- 这个项目有很多“结构化数据”，例如电影信息、版本信息、平台结果、判定标签
- TypeScript 能减少后期字段混乱和逻辑误判

#### Tailwind CSS

- 足够快，适合先做一个清晰实用的内部工具
- 第一版重点不在复杂视觉，而在信息密度和可读性

#### Playwright

- 这个项目很可能需要进入动态页面甚至播放器环境获取真实时长
- Playwright 比纯 HTTP 抓取更适合处理腾讯视频、B 站这类站点
- 后续需要模拟点击、等待渲染、读取页面数据时也更灵活

#### Cheerio

- 对静态 HTML 解析很轻量
- 可以和 Playwright 配合使用：先拿页面，再抽取结构化信息

#### 本地文件缓存

- 第一版先不引入数据库，降低复杂度
- 很适合你这种“常查一些电影、给自己和朋友用”的场景
- 后续如果数据量变大，再切数据库也不晚

## 13.5 第一版目录规划建议

建议第一版采用下面这种结构：

- `app/`
- `app/page.tsx`
- `app/api/search/route.ts`
- `app/api/analyze/route.ts`
- `lib/douban/`
- `lib/platforms/tencent/`
- `lib/platforms/bilibili/`
- `lib/runtime/`
- `lib/judgement/`
- `lib/cache/`
- `types/`
- `data/cache/`

### 模块职责

- `app/page.tsx`
  - 输入电影名或豆瓣链接
  - 展示搜索结果和分析结果

- `app/api/search/route.ts`
  - 接收电影名
  - 搜索并返回候选电影

- `app/api/analyze/route.ts`
  - 接收确认后的电影信息
  - 拉取豆瓣信息、平台信息、版本信息
  - 输出最终判定结果

- `lib/douban/`
  - 豆瓣搜索、详情页解析、播放源解析、版本时长提取

- `lib/platforms/tencent/`
  - 腾讯视频页面识别和时长提取

- `lib/platforms/bilibili/`
  - B 站页面识别和时长提取

- `lib/judgement/`
  - 参考版本选择
  - 时长差值比较
  - 标签输出

- `lib/cache/`
  - 本地 JSON 缓存
  - 减少重复抓取

- `types/`
  - 电影、版本、平台结果、最终报告的类型定义

## 13.6 第一版缓存策略建议

为了减少反复抓站点，第一版建议做本地缓存。

缓存对象包括：

- 电影基础信息
- 豆瓣详情页解析结果
- 已识别版本信息
- 腾讯视频分析结果
- B 站分析结果

### 缓存形式

- 存成 `data/cache/*.json`

### 缓存策略

- 普通电影信息缓存 `7 天`
- 平台时长缓存 `1 到 3 天`
- 用户手动触发“重新检测”时跳过缓存

## 13.7 为什么第一版不建议做浏览器插件

- 现在最难的是“判定逻辑”和“抓取稳定性”，不是入口形式
- 先做网页工具，调试抓取和判定都会更容易
- 等网页版本稳定后，再把结果注入豆瓣页面会更顺

## 13.8 第一版运行方式

建议第一版先以本地开发模式运行：

- 你在自己电脑启动服务
- 你和朋友通过本地访问或后续部署访问
- 等规则稳定后，再决定是否部署到线上

## 13.9 当前技术结论

第一版采用：

- `Next.js + TypeScript + Tailwind CSS`
- `Playwright + Cheerio`
- `本地文件缓存`
- `网页工具形态`

这是当前最适合你这个项目阶段的组合，重点是先把“输入电影 -> 输出无删减判断”这条链路跑通。

## 14. 平台接入策略建议

第一批建议优先验证：

- 腾讯视频
- 哔哩哔哩

原因：

- 这是你第一版最关心的两个平台
- 两个平台定位差异较大，适合尽早暴露判定规则问题
- 腾讯视频更接近主流长视频平台，哔哩哔哩更容易出现版本差异、活动页和合集页等特殊情况，适合尽早验证边界

第二批再考虑：

- 爱奇艺
- 优酷
- 咪咕视频
- 芒果 TV

## 15. 结果展示建议

每个平台至少显示：

- 平台名
- 是否可播放
- 抓到的片长
- 与豆瓣差值
- 与当前参考版本差值
- 判定标签
- 判定理由
- 跳转链接

电影基本信息区域至少显示：

- 中文名 / 原名
- 年份
- 导演
- 豆瓣评分
- 已识别版本与对应片长
- 当前参考版本

### 判定理由示例

- `平台时长 107:12，与豆瓣 107 分钟基本一致，处于正常误差范围内`
- `平台时长比豆瓣短 6 分 14 秒，疑似存在删减`
- `平台时长比豆瓣长 11 分钟，更像是不同发行版本而非删减`
- `平台页面无法获得真实片长，暂时无法判断`
- `当前优先参考导演剪辑版 128:04；平台时长 127:20，差异 44 秒，属于正常误差`
- `已检测到中国大陆版 118 分钟、电影节版 124 分钟；平台更接近中国大陆版，可能并非完整版本`

## 16. 成功标准

MVP 达成标准：

- 用户输入一部常见电影后，能稳定拿到豆瓣信息
- 结果中能展示导演和已识别版本信息
- 能优先判断腾讯视频和哔哩哔哩是否存在明显删减
- 系统能输出可解释的标签，而不是只给结论
- 对秒级和一分钟内差异不误判

## 17. 当前建议的开发顺序

1. 先做 PRD 和判定规则。
2. 再做“单电影查询”的最小原型。
3. 第一版只接入豆瓣 + 腾讯视频 + 哔哩哔哩。
4. 跑一些你熟悉的样本电影，人工验证误判情况。
5. 再决定要不要做插件入口、登录态支持、历史记录。

## 18. 当前最需要确认的问题

### 产品决策

- 你更想先做“自己用得爽”的工具，还是以后也想给别人公开使用？
- 你是否接受第一版只支持腾讯视频和哔哩哔哩？

### 规则决策

- 你能接受初版把 `<= 90 秒` 视为正常误差吗？
- 如果豆瓣和平台差了 `2 到 4 分钟`，你希望默认显示“可能删减”还是“版本不一致待确认”？

### 采集决策

- 如果某平台只有登录后才能拿到真实片长，第一版是否接受直接标记“信息不足”？

## 19. 我的当前建议

如果我们要稳一点，我建议这样开工：

- 先做 **网页 MVP**
- 先支持 **豆瓣 + 腾讯视频 + 哔哩哔哩**
- 先把 **判定逻辑和误差容忍**做对
- 把“信息不足”和“版本不一致”当成正式结果，而不是硬判

这样第一版最容易做出来，也最不容易把用户带偏。
