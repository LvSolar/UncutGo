# FindMyMovie PRD

## 1. 项目一句话

输入一部电影，自动整合豆瓣与国内播放源信息，告诉用户“去哪里更可能看到无删减版”。

## 2. 当前定位

FindMyMovie 当前不是面向大众的开放平台，而是先服务你和少量朋友稳定使用的网页工具。

它的核心价值不是做一个泛影视搜索产品，而是把下面这条手动流程自动化：

1. 去豆瓣查评分、导演、简介和片长
2. 看国内哪些平台有资源
3. 逐个平台比对时长
4. 判断哪里更可能是无删减版

## 3. 核心目标

### 用户目标

- 输入电影名后，快速知道国内哪些平台更可能是无删减版
- 同时看到这个判断背后的版本依据和平台时长依据
- 遇到无播放源或信息不足时，也得到明确结果，而不是模糊失败

### 产品目标

- 把“搜电影 + 查播放源 + 对比时长 + 输出判断”自动化
- 降低分钟取整、版本差异、片头片尾差异造成的误判
- 优先保证结果可信，再考虑覆盖更多平台和更花哨的交互

## 4. 当前明确规则

### 4.1 多版本规则

- 一部电影可能存在多个合法版本，结果里应展示多个版本时长
- 判断删减时，优先参考：导演剪辑版 > 电影节版 > 非中国大陆版 > 其他国际版 > 中国大陆版
- 同优先级多个版本并存时，默认选更长的版本作为参考

### 4.2 豆瓣片长规则

- 豆瓣分钟显示不等于精确秒数
- 像 `1:23:20` 显示成 `84 分钟` 是正常情况
- 不能因为分钟取整而直接误判删减

### 4.3 平台规则

当前优先支持：

- 腾讯视频
- 哔哩哔哩
- 爱奇艺
- 优酷
- Libvio

### 4.4 时长判断规则

- 平台比参考版本略短时，要比“略长”更谨慎
- 平台比参考版本略长时，需要容忍片头片尾、平台提示、黑场等常见附加内容
- 平台时长明显更长时，更优先判断为版本不一致，而不是直接视作无删减

### 4.5 正式结果规则

以下都属于正式结果，不是异常：

- 大概率无删减
- 可能无删减
- 版本不一致
- 可能删减
- 疑似删减
- 信息不足
- 无国内播放源

## 5. 当前产品能力

### 已完成

- 真实豆瓣搜索
- 真实豆瓣详情页解析
- 豆瓣播放源识别
- 腾讯视频真实时长抓取
- 哔哩哔哩真实时长抓取
- 爱奇艺真实时长抓取
- 优酷真实时长抓取
- Libvio 站内搜索与播放页时长抓取
- 多版本参考规则与删减判断
- 本地文件缓存
- 网页原型与 API 打通
- 结果页状态表达优化

### 当前输入

- 电影名
- 豆瓣电影链接

### 当前输出

- 豆瓣电影基础信息
- 导演
- 豆瓣评分
- 豆瓣简介
- 多个已识别版本时长
- 当前参考版本
- 豆瓣识别到的国内播放源
- 各平台抓取到的真实时长
- 最终判定标签
- 判定理由
- 缓存或实时状态
- 平台当前状态说明

## 6. 当前判定标签

- `大概率无删减`
- `可能无删减`
- `版本不一致`
- `可能删减`
- `疑似删减`
- `信息不足`

### 当前标签含义

- `大概率无删减`：与当前参考版本差异处于正常误差范围，或略长但仍像平台附加内容
- `可能无删减`：比参考版本略短，或略长但还不明显像另一发行版本
- `版本不一致`：更像是另一个合法发行版本，而不是单纯删减
- `可能删减`：比参考版本短得比较明显，但还不到特别极端
- `疑似删减`：比参考版本短很多，疑似明显删减
- `信息不足`：暂时拿不到可信的正片时长，无法稳定判断

## 7. 当前核心流程

1. 用户输入电影名或豆瓣链接
2. 系统定位豆瓣电影
3. 系统抓取豆瓣详情页
4. 系统提取电影基础信息、版本时长、播放源链接
5. 系统逐个平台抓取真实时长
6. 系统选出当前参考版本
7. 系统与各平台时长做统一比较
8. 系统输出标签、理由和平台状态说明
9. 系统缓存分析结果

## 8. 当前技术策略

### 当前实现方式

- 前端：Next.js 页面
- 后端：Next.js Route Handlers
- 语言：TypeScript
- 页面解析：Cheerio
- 缓存：本地 JSON 文件缓存
- Libvio 特殊兜底：浏览器级渲染读取播放页时长

### 当前缓存策略

- 分析结果按缓存版本写入 `data/cache`
- 当抓取逻辑变化时，通过提升缓存版本避免旧结果误导

### 代理注意事项

本机启用了 Clash Verge。后续遇到联网问题时，需要优先考虑代理因素；当前已知常见端口约为 `7897`。

## 9. 当前边界

### 当前不做

- 不判断画质、字幕质量、码率
- 不聚合盗版资源导航站
- 不承诺法律或发行层面的“官方无删减”结论
- 不在当前阶段做大众开放平台能力
- 不优先做账号、社区、推荐流等功能

### 当前风险点

- 平台页面结构会变
- 某些平台拿到的是详情页信息，不一定总能进入播放器上下文
- Libvio 这类站点部分时长信息由浏览器运行后动态生成
- 同一电影的多版本问题仍然会影响判断可信度
- 部分电影会出现无播放源或平台信息不足

## 11. 下一阶段优先级

### 第一优先级

- 补强豆瓣搜索 fallback，减少别名、简称或中英混合标题搜不到的情况

### 第二优先级

- 继续补强版本信息提取
- 不只拿到单一片长，而是尽量保留多个版本线索

### 第三优先级

- 做部署准备，方便直接发网址给朋友使用
- 之后再考虑正式部署到线上

## 12. 成功标准

当前阶段的成功标准是：

- 常见电影能稳定走完“搜索 -> 分析 -> 输出判断”流程
- 腾讯、B 站、爱奇艺、优酷和 Libvio 能给出尽量可信的时长结果
- 用户能看懂系统为什么给出这个判断
- 无播放源、信息不足、版本不一致等情况都有明确表达
- 新增或修复抓取逻辑后，可以通过缓存版本升级避免旧缓存误导

## 13. 当前结论

FindMyMovie 当前已经从“可运行原型”进入“可持续迭代的小工具”阶段。

接下来最重要的不是继续堆更多功能，而是：

- 提高搜索稳定性
- 继续增强版本依据
- 为部署和分享做好准备
